# Adversarial Review — AceFluency Practice-English-with-Peers Rebuild

*Reviewed against `prd.md` (status: final, 2026-06-09) and `addendum.md`. Previous review superseded — OQ 1 (SSLCommerz), OQ 6 (LiveKit Cloud), and FR-20 (refund mechanism) are now resolved in the current PRD.*

## Summary

This PRD is unusually well-structured for a rebuild spec, but it carries three systemic risks that a friendly reviewer would wave past: (1) it scopes a full billing/refund/moderation system from scratch inside a "rebuild" that also rewires auth, calls, and matching — the aggregate scope is a new product, not a patch, and no timeline or team-size constraint is stated anywhere; (2) its success metrics are disconnected from the user journeys that justify the rebuild, so the team can hit every SM and still not fix the problems that motivated the work; and (3) the India market is acknowledged in screenshots and pricing but treated as a bolt-on afterthought, creating a silent second product surface that will double payment, compliance, and support work without any corresponding scope adjustment.

## Findings

### Critical

- **[critical]** Scope is unbounded — no timeline, no team size, no phasing (§0, §6) — The PRD defines 20 functional requirements spanning six feature areas, including two greenfield subsystems (billing and moderation) plus a payment provider integration, but never once states a delivery timeline, team size, or phase boundary. §6 "MVP Scope" literally includes *every single FR*. If everything is MVP, nothing is MVP. The word "rebuild" implies a bounded effort; the spec describes a ground-up product launch. *Fix:* Define a hard phase-1 that ships in ≤8 weeks with ≤5 FRs (e.g., auth + call reliability only), and a phase-2 for billing/moderation. State team size. State a date or a sprint count.

- **[critical]** India market is silently in scope but unscoped (§4.5, OQ 9, Addendum §H) — The PRD confirms the live app serves India with INR pricing and Indian tutors. OQ 9 adds a second PSP. But no FR addresses India-specific requirements: UPI payment flows, GST invoicing, RBI auto-debit mandate (RBI's 2021 e-mandate rules require explicit re-consent for recurring payments — this is not optional). The refund matrix in Addendum §B is "written for SSLCommerz (BDT)" and the addendum hand-waves "the same matrix applies but execution uses the India PSP's refund API." That is not a requirement; that is a wish. India is either in MVP or out; the PRD says neither. *Fix:* Explicitly declare India in-scope or out-of-scope for MVP. If in, add FRs for UPI, e-mandate compliance, GST invoicing, and the India PSP integration. If out, say so and remove INR references.

- **[critical]** Success metrics do not validate the user journeys (§7 vs. §2.3) — UJ-1 (Sumit) is about a paying user getting banned for leaving bad calls. The metric that would validate fixing this is "false-positive strike rate" or "% of cooldowns issued to users who reported a partner." No such metric exists. SM-1 (disconnect rate ≤5%) measures network disconnects, not moderation-caused disconnects. UJ-4 (Hari) is about a paid filter not being honored. SM-9 (filter fulfillment ≥90%) allows a 10% violation rate on a *paid* feature — that means 1 in 10 matches can violate a filter the user paid for, and the PRD considers that success. UJ-5 (Mridu) is about auto-renewal with no escape. No SM measures "% of users who successfully cancelled auto-renew in-app." *Fix:* Add SM for false-positive moderation rate (target: ≤1%), paid-filter violation rate (target: 0% — a paid filter is a contract, not a best-effort), and cancellation completion rate (already partially in SM-10 but framed as flow completion, not actual billing outcome).

### High

- **[high]** The Skip button is load-bearing but entirely unowned in the PRD (§4.4 FR-11, Addendum §C) — FR-11 says: "Skip is a first-class in-call action… does not count as a strike." Addendum §C says: "The Skip button is the load-bearing piece. Without it, the only way a user can reject a bad match is to hang up, and hanging up counts as a strike." But the PRD defers Skip to `bmad-ux` downstream with no FR, no acceptance criteria, no screen, no behavior spec. A "load-bearing piece" that has no requirement is a structural defect. If `bmad-ux` deprioritizes or redesigns Skip, the entire moderation system collapses back to the failure mode UJ-1 describes. *Fix:* Promote Skip to a full FR with testable consequences: what happens to the current call, does the user re-enter the queue automatically, is there a cooldown between Skips, does the skipped partner get notified?

- **[high]** FR-20 refund auto-deny rule is self-contradictory (§4.5 FR-20 vs. Addendum §B) — FR-20 states auto-deny triggers include "account has >5 completed sessions." Addendum §B states auto-deny triggers include "≥1 completed session." These contradict each other. Under the addendum rule, a user who completed one 5-minute call and then experienced 72 hours of login failures would be auto-denied a refund — which is exactly UJ-5's failure mode. The PRD wrote a refund system that would deny Mridu's refund. *Fix:* Reconcile the two documents. The auto-deny threshold should be high enough that a user who barely used the product is not penalized. Suggest: auto-deny only if >5 completed sessions AND refund requested >14 days after charge (both conditions).

- **[high]** No FR for the post-call rating flow, despite confirming it exists and depending on it (§2.3, OQ 7, Addendum §D, §H) — The addendum confirms the post-call rating flow exists in the live app, OQ 7 says the rebuild "must preserve this flow" and "ensure the rating data flows into the matching service," and Addendum §D lists five quality inputs that depend on it. But there is no FR for the post-call rating. It is not in §4. It is not in §6 MVP scope. It is an orphaned dependency: match quality depends on it, the moderation system's "short disconnect" logic benefits from it, and it is silently assumed to survive the rebuild unchanged. *Fix:* Add an FR (e.g., FR-21) for post-call rating: preserve existing flow, add "Did this partner help you practice?" question, feed rating into `matchPartners` scoring, define retention policy.

- **[high]** "Rebuild" vs. reality: the PRD ignores the existing app's feature surface (§5, Addendum §H) — Addendum §H reveals the live app has 6 call types, a tutor marketplace, AI speaking tests, recorded courses, drama tab, rooms/voice clubs, daily speaking streaks, a friend graph, and communities. The PRD's §5 Non-Goals says these are "not modified in this rebuild." But the PRD also rewires auth (§4.1), the call system (§4.2), the match service (§4.3), the moderation system (§4.4), and adds billing from scratch (§4.5). Every one of those existing features depends on auth, calls, or moderation. The PRD does not address regression risk: if the rebuild changes the auth token format, does the tutor marketplace break? If the call lifecycle changes, do group classes still work? If moderation state changes, does the community guidelines modal still function? *Fix:* Add a §4.7 "Regression Scope" FR that explicitly lists which existing features must continue working unchanged and defines acceptance tests for them.

- **[high]** SSLCommerz refund API failure has no fallback besides a 7-day SLA (§4.5 FR-20) — FR-20 says "All auto-approve refunds post a refund record to SSLCommerz first, then mark the user's billing record as refunded locally. Failed refund posts fall through to the human-review path." This means a SSLCommerz API outage converts every auto-approve refund into a human-review case with a 7-day SLA. During a payment provider outage, the refund system degrades from "24 hours" to "7 days" with no user communication. *Fix:* Define a retry policy (e.g., 3 retries over 24h before fallback to human review). Add a user-visible state: "Refund approved — processing with payment provider" so the user knows the decision was made even if the money hasn't moved.

### Medium

- **[medium]** SM-2 baseline is fabricated (§7, §9) — SM-2 says the current app-store rating is "2.0–3.5 in sampled reviews" and targets ≥4.0. §9 Assumptions Index says "the actual store rating is unknown." The PRD set a success metric against a baseline it does not know. If the actual current rating is 3.8, a target of 4.0 is trivial. If it's 1.5, 4.0 may be unreachable in one release. *Fix:* Measure the actual current Play Store rating before finalizing SM-2. State the real baseline.

- **[medium]** FR-3 phone OTP has no cost model or abuse mitigation (§4.1 FR-3) — OTP via SMS in Bangladesh and India is not free. At scale, SMS costs are material (₹0.15–0.25/SMS in India, higher in BD). The PRD specifies "voice call fallback for unreachable numbers" which is 3–5x more expensive. No FR addresses: rate-limiting OTP requests, CAPTCHA before OTP send, cost ceiling per user, or abuse prevention (an attacker can burn SMS budget by requesting OTPs at scale). *Fix:* Add an FR or NFR for OTP rate-limiting (e.g., max 5 OTP requests per phone per hour), and require architecture to define the cost model.

- **[medium]** "Human review" appears in three places with no definition of who, when, or how (§4.4 FR-11, §4.5 FR-16, §4.5 FR-20) — FR-11 says "automatic review by a human moderator" at 10 disconnects. FR-20 says "Human review: everything else. SLA: 7 days." FR-16 notes the 24h SLA "assumes the team has committed to a support rota, or that an AI-assisted tier-1 response path exists." Three separate systems require human review, but the PRD never states: how many humans, what tools they use, what their working hours are, or whether the team has committed to staffing this. The PRD is writing checks the org may not be able to cash. *Fix:* Add a cross-cutting organizational dependency section. State explicitly: "This PRD requires N support/moderation staff or an AI triage system. Without this, FR-11, FR-16, and FR-20 SLAs are undeliverable."

- **[medium]** Gender filter fulfillment target of 90% means 10% paid-feature violation (§7 SM-9) — A user who pays for gender filtering and gets a wrong-gender match 10% of the time is experiencing the exact failure described in UJ-4. SM-9 at 90% is not a success metric; it is a formalization of partial failure. The PRD acknowledges the empty-pool problem but does not adjust the metric to distinguish "no match available, user shown honest wait state" from "user matched with wrong gender." *Fix:* Redefine SM-9: 0% of matches should violate the filter. Separately measure "% of gender-filtered queue sessions that result in no match within 90s" to track pool health.

- **[medium]** No accessibility requirements anywhere (§4, §6) — The PRD targets South Asian users on low-end Android devices, many of whom may have limited digital literacy. No FR mentions: text size, contrast ratios, screen reader support, RTL text support (Urdu is listed as a native language in FR-9), or localization. The app serves at least two languages (Bangla, Hindi) but no FR addresses UI localization. *Fix:* Add a cross-cutting NFR for WCAG 2.1 AA compliance on primary flows, and an FR for UI localization (at minimum: English, Bangla, Hindi).

- **[medium]** FR-8 empty-pool fallback creates a dark pattern risk (§4.3 FR-8) — FR-8 states: "if no valid match is available for 5 minutes, the system offers the user the option to (a) keep waiting, or (b) drop the gender filter for the current session." This is the correct UX. But in practice, if the pool is consistently empty (e.g., very few female users), the user will be asked to drop their paid filter every single session. This is a de facto upsell-for-nothing: the user pays for a filter that functionally does not work because the pool doesn't support it. No FR addresses what happens when a filter is *structurally* unfulfillable. *Fix:* Add a metric for "% of gender-filtered sessions that resolve without fallback prompt." If this drops below 50% for 7 consecutive days, trigger a PM review of whether the filter should be sold at all.

- **[medium]** Addendum §B auto-deny for "≥1 completed session" is hostile to users (Addendum §B) — Under this rule, a user who completed one 5-minute call on day 1, then experienced 6 days of crashes, would be auto-denied a refund. This is the opposite of the PRD's stated philosophy ("make every one of those failure moments not happen"). The auto-deny threshold is set so low that it will generate the exact 1-star reviews the rebuild is trying to prevent. *Fix:* Raise the auto-deny threshold to ≥5 completed sessions, or make the auto-deny conditional on the absence of any reported technical issues.

### Low

- **[low]** CEFR placement is assumed accurate but never validated (§3, §4.3) — The matching system filters by CEFR ±1, but the PRD never describes how CEFR is assigned. Addendum §D mentions "Pre-call level re-prompt" as a deferred v2 input. If CEFR placement is inaccurate (self-reported), the entire matching system is filtering on noise, and "bad matches" will persist regardless of the new filter infrastructure. *Fix:* Acknowledge CEFR accuracy as a known risk, and add a v1 metric: "% of post-call ratings where the user selected 'partner was too advanced/too basic'."

- **[low]** FR-12 report-voids-strike is exploitable (§4.4 FR-12) — FR-12 says: "Report during or within 60s after a call automatically voids the strike for the reporting user." A user who wants to serial-disconnect without consequence can report every partner for "non-participation" within 60s. The PRD does not address false-report abuse. *Fix:* Add a counter-metric: if a user reports >50% of their partners in a 24h window, flag the user for review instead of voiding all strikes.

- **[low]** No data migration plan for existing users (§4.5, §4.4) — The rebuild adds new schema fields (`gender`, `nativeLanguage`, moderation state, billing tables) and changes auth behavior. Existing users presumably have accounts. No FR addresses: how existing users get the new fields populated, whether they see an onboarding re-prompt, or how their existing subscription (if managed outside the app, e.g., via Play Store billing) maps to the new billing system. *Fix:* Add an FR or architecture note for data migration: existing users get a one-time onboarding prompt for new fields, existing subscriptions are reconciled with the new billing table.

- **[low]** "Pronunciation Score" is in the glossary but appears in zero FRs (§3) — The glossary defines "Pronunciation Score" as "a per-clip numeric score (0–100) produced by the existing on-device pronunciation model." No FR references it. No SM validates it. It is a glossary ghost — it exists in the spec but has no functional home. *Fix:* Either remove from glossary or add to §6.2 Out of Scope with a note that it is preserved as-is from the existing app.

- **[low]** No error/edge-case handling for dual-PSP billing (§4.5, OQ 9) — The PRD requires SSLCommerz for BD and a TBD provider for India, but no FR addresses: what happens if a user switches countries (BD SIM → IN SIM), how currency is determined, whether a user can have two active subscriptions (one per PSP), or how billing history displays cross-currency transactions. *Fix:* Add an FR or architecture constraint: currency is determined at purchase time and locked for the billing period; users cannot hold parallel subscriptions across PSPs.
