# Adversarial Review — AceFluency Personalized Content Recommendation Engine

## Summary

This PRD describes a recommendation engine that is 40% broken at the scoring layer (random embeddings), invisible to 90%+ of users (no mobile feed), and incapable of accepting the personalization inputs it claims to use (no preferences UI). The document is unusually honest about these gaps, but then contradicts itself by labeling the system "fully implemented" in multiple places. The MVP gap table is the real PRD — the rest is a spec for a demo, not a product.

## Findings

- **Critical** — Scoring engine runs on random noise (§4.2 FR-6, §8.3) — The PRD states content embeddings are "random 384-dim vectors (placeholder)" from the seed endpoint, yet the hybrid scoring pipeline (§4.3 FR-8) weights cosine similarity at 40% — the single largest signal. The system is currently making recommendations where nearly half the scoring weight is mathematically random. The PRD marks this as "⚠️ Partially implemented" — it is not partially implemented, it is actively producing garbage output that degrades user trust. Every success metric in §7 (SM-R1, SM-R3, SM-R4) is unmeasurable against a system serving noise-contaminated scores. *Fix:* Reclassify FR-6 as a hard blocker. No recommendation feed should be user-facing until real embeddings are computed. Add a feature flag to disable the feed entirely until FR-6 is resolved.

- **Critical** — Profile embeddings are also fake (§10, last bullet) — The Assumptions Index casually drops that `recomputeEmbedding` in `models.ts` uses *hardcoded placeholder values* ("CEFR: B1, interests: [], goals: [], native: Bangla, age: 25") instead of the user's actual profile. This means the 40%-weighted cosine similarity signal is not just random on the content side — it is also a constant on the user side. Every user gets the same profile embedding. The recommendation engine has **zero personalization from embeddings** for both sides of the similarity computation. This is buried in the last line of §10 instead of being called out as a critical gap in §6.2. *Fix:* Add this to the §6.2 gap table as a **High/Blocker** item. The profile embedding pipeline must read real user data before any embedding-based scoring is meaningful.

- **Critical** — No mobile surface for 90%+ of users (§4.8, §5) — The PRD acknowledges "90%+ of active users are on mobile" (citing parent PRD §5), then reveals the recommendation feed exists only on web. The PRD correctly flags this as MVP-required (FR-23), but the §5 Non-Goals section contradicts itself: it first lists "Native mobile recommendation feed" as a non-goal header, then immediately walks it back by saying it's actually MVP-required. This is confusing and dangerous — a reader scanning §5 will conclude mobile is out of scope. *Fix:* Remove the mobile feed from §5 entirely. It is not a non-goal; it is a blocker. The §5 entry should only cover "admin tools on mobile" as the actual non-goal.

- **High** — No preferences UI means 30% of scoring is dead weight (§4.5, FR-17) — The PRD identifies that tag overlap (20%) and content type preference (10%) signals require user preferences, then acknowledges no web or mobile UI exists for users to set preferences. The API exists but is unreachable by real users. Combined with the broken embedding signal (40%), this means **70% of the scoring algorithm receives either garbage or empty inputs**. Only the CEFR closeness signal (30%) works correctly. The PRD phrases this as "degrading recommendation quality" — it should say "the recommendation engine is a CEFR filter with extra steps." *Fix:* Elevate FR-17 to blocker status. Add a preferences onboarding flow to the user's first session (not just a settings page) to ensure preference data is collected proactively.

- **High** — Success metrics cannot validate the thesis (§7) — SM-R1 targets "≥30% engagement rate within 30 days of launch," but the scoring engine is fundamentally broken (random embeddings, fake profile embeddings, empty preferences). Hitting 30% engagement would validate nothing about the recommendation algorithm — it would only prove users click on things when shown a list. Conversely, missing the target would be blamed on the algorithm when the real cause is garbage inputs. None of the success metrics have a control condition (e.g., "compared to a random feed" or "compared to a CEFR-only feed"). *Fix:* Add a baseline comparison: measure engagement on a simple "newest content at your CEFR level" feed vs. the hybrid-scored feed. Without this, SM-R1 through SM-R4 are vanity metrics.

- **High** — Excluding all interacted content creates a depletion spiral (§4.3 FR-9, §10) — The candidate filter excludes *any* content the user has interacted with (view, like, bookmark, complete, share, dismiss). With only 12 seed items (2 per CEFR level) and a ±1 CEFR filter, a B1 user has at most 6 candidate items. After viewing all 6, the recommendation feed is permanently empty. The PRD's §10 Assumptions notes this might be wrong for bookmarks/likes, but does not model the depletion scenario. Even with a larger library, heavy users will exhaust their CEFR band quickly. *Fix:* Change the exclusion logic: exclude only `dismiss` and `complete` interactions from the candidate pool. Allow `view`, `like`, `bookmark`, and `share` items to reappear (perhaps with a decay penalty). Add an explicit "empty state" design for when candidates are exhausted, including a prompt to try a different CEFR level or content type.

- **High** — `createContent` is `Protected`, not `Admin` — security gap (§8.2) — The API surface table lists `createContent` as "Protected" (authenticated user), not "Admin." This means any logged-in learner can create content items. The PRD describes content creation as an admin activity (§4.1: "Admins and content curators"), but the implementation does not enforce this. A malicious or mischievous user could flood the content library. *Fix:* Gate `createContent` behind `adminProcedure` (same as `adminDeleteContent`). If user-generated content is planned for the future, make it a separate, moderated endpoint.

- **High** — Public seed endpoint is a wipe-the-database button (§4.7 FR-22, §8.2) — The seed endpoint is `publicProcedure` and uses `onConflictDoUpdate`. Anyone on the internet can call it and overwrite existing content items, user preferences, and interaction data with demo values. The PRD flags this as "security concern" in §4.7 and Open Question 1, but classifies it as **Medium** priority in §6.2. An unauthenticated endpoint that can mutate production data is not medium — it is a critical production safety issue. *Fix:* Reclassify to High/Blocker. Remove the endpoint entirely in production builds, or gate it behind `adminProcedure` + an environment check (`NODE_ENV !== 'production'`).

- **Medium** — Hybrid score weights are arbitrary and unvalidated (§4.3 FR-8, §10) — The weights (0.4 / 0.3 / 0.2 / 0.1) are stated as "reasonable starting values" with a §10 assumption that they "should be tuned from engagement data." But there is no mechanism to tune them — no A/B testing (explicitly a non-goal in §5), no weight configuration endpoint, no admin UI to adjust weights, and no analytics that correlate weight changes with engagement changes. The weights are hardcoded constants with no path to optimization. *Fix:* At minimum, externalize the weights to environment variables or a config table so they can be changed without a code deploy. Add a §9 Open Question about the tuning methodology.

- **Medium** — FR numbering is broken — FR-22 in §4.7 appears after FR-23 in §4.8 (§4.7, §4.8) — Section 4.8 (Native Mobile) defines FR-23 and appears *before* §4.7 (Seed & Demo Data) which defines FR-22. The section numbering itself skips from §4.6 to §4.8 and then back to §4.7. This is a document structure error that will confuse downstream consumers (epic/story creation, architecture docs). *Fix:* Reorder sections: §4.7 should be Seed & Demo, §4.8 should be Native Mobile, and FR numbers should be sequential.

- **Medium** — No content consumption tracking — only metadata interactions (§4.4) — The interaction types include `view` and `complete`, but the PRD never specifies what constitutes a "view" (page load? 3 seconds? scroll to bottom?) or a "complete" (watched 100%? clicked "done"?). The content items link to external source URLs — the system has no way to know if a user actually consumed the content or just saw the card. SM-R4 (like-to-view ratio) is meaningless if "view" means "card appeared on screen." *Fix:* Define "view" precisely (e.g., "user clicked through to content detail page or opened source URL"). Define "complete" (e.g., "user explicitly marked as complete via button"). Add these definitions to §3 Glossary.

- **Medium** — Score caching creates stale-on-write problem (§4.3 FR-10, FR-16) — When a user updates their preferences (FR-16), the cached recommendation scores (24h TTL) are not invalidated. The PRD states: "Updated preferences take effect on the next recommendation recalculation (not retroactively applied to cached scores)." This means a user who sets their interests for the first time will see *zero change* in their feed until they manually hit "Recalculate" or wait 24 hours. This directly contradicts the "feels like the app knows me" emotional JTBD (§2.4). *Fix:* Invalidate cached scores when preferences change. Or at minimum, auto-trigger a recalculation after a preference update.

- **Medium** — Parent PRD dependency is unversioned (§0, §8.4) — The PRD references `prd-community-2026-06-09` as the parent but does not pin a version or hash. If the parent PRD is updated (e.g., CEFR placement logic changes, partner matching weights shift), this PRD's §8.4 shared infrastructure assumptions silently break. The decision log has only one entry and no record of cross-PRD dependency management. *Fix:* Add a version/hash reference to the parent PRD. Add a §10 assumption: "Parent PRD §4.3 matchmaking algorithm and CEFR placement schema remain stable."

- **Medium** — §6.2 and §6.3 contradict on FR-4 (Update content item) — §6.2 lists FR-4 as "Medium" priority for MVP completion. §6.3 lists FR-4 again as "Medium" priority for v2 roadmap. FR-4 cannot be both "must complete before MVP done" and "out of scope for MVP." *Fix:* Pick one. If the delete-and-recreate workaround is acceptable for MVP, move FR-4 entirely to §6.3. If not, keep it in §6.2 and remove it from §6.3.

- **Low** — Candidate pool cap of 200 is arbitrary (§4.3 FR-9) — The candidate pool is capped at 200 items "as a performance guard," but no benchmarking data justifies this number. With a CEFR ±1 filter and a small content library, the cap is irrelevant. With a large library (1000+ items per CEFR band), the cap could exclude relevant content. The selection of *which* 200 items to keep is not specified — is it random? Newest? *Fix:* Specify the ordering used when truncating to 200 (e.g., `createdAt DESC`). Add a §9 Open Question about whether 200 is the right cap and how it should scale.

- **Low** — No accessibility or i18n considerations for the recommendation feed (§4.3, §4.8) — The PRD specifies card layouts, badges, and interaction buttons without any mention of screen reader support, keyboard navigation, or right-to-left language support. The target users are Bengali speakers — the UI copy language (English vs. Bengali) for the feed is unspecified. *Fix:* Add a paragraph to §4.3 or a cross-cutting NFR section addressing WCAG 2.1 AA compliance and UI language for the recommendation feed.

- **Low** — Decision log is skeletal (`.decision-log.md`) — The log has a single entry from June 12 recording the initial feature selection. The PRD was updated on June 13 and contains numerous design decisions (scoring weights, TTL values, CEFR ±1 filter width, interaction exclusion logic, candidate cap) — none of which are logged. The decision log is not fulfilling its purpose. *Fix:* Backfill the decision log with the design decisions made during the PRD authoring session, including the rationale for each weight and threshold value.
