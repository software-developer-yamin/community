AceFluency Recommendation Engine — Validation Report

prd-community-recommendation-engine-2026-06-12/prd.md
Good — 7/7 Strong

This is a genuinely strong companion PRD — one of the better reverse-engineered-from-code specs. The feature inventory is code-validated, gaps are called out with FR numbers and priority/effort, the scoring algorithm is specified to testable precision, and the success metrics actually measure the thesis. The rubric review rates all 7 dimensions strong.

The adversarial review reveals a devastating gap the rubric would not catch: the recommendation engine is 40% broken at the scoring layer. Content embeddings are random placeholders, profile embeddings are hardcoded constants, and no preferences UI exists — meaning 70% of the hybrid scoring algorithm receives garbage or empty inputs. The system is currently a CEFR filter with extra steps. Additionally, the recommendation feed has no mobile surface despite 90%+ of users being on mobile.

Combined finding counts: 3 critical, 5 high, 6 medium, 5 low. The critical findings cluster around the broken scoring pipeline (random embeddings on both sides) and the missing mobile surface. The high findings target the dead-weight preferences signal, unvalidatable success metrics, content depletion spiral, security gaps, and the public seed endpoint.

Decision-readiness

Strong

The PRD is structured so a PM or engineering lead can act immediately. §6 MVP Scope splits cleanly into "already implemented" (§6.1), "must complete before MVP done" (§6.2 gap table with priority, effort, and blocker status), and "v2 roadmap" (§6.3). Strategic coherence

Strong

The PRD has a clear thesis: the recommendation engine is the "training program" that makes peer voice practice more effective by filling between-session time with level-appropriate content. Features serve the thesis, success metrics validate it, and counter-metrics (SM-RC1 feed homogeneity, SM-RC2 CEFR level lock-in) catch the failure modes.

The relationship to the parent PRD is well-articulated. The recommendation engine and partner matching share the profile embedding pipeline, CEFR placement, and BGE model — one user model serving two surfaces.

Medium

No metric validates the core thesis — rec-fed learners practice better

§7
SM-R1–R4 measure recommendation engagement, but the strategic claim is that recommendations improve the core product (voice practice). No metric correlates feed engagement with call completion rates or practice frequency.
Fix: Add a stretch metric: "Correlation between recommendation feed engagement and voice practice frequency. Hypothesis: users who interact with ≥3 recommended items per week have ≥15% higher call completion rates."

Adversarial review

review-adversarial.md

This PRD describes a recommendation engine that is 40% broken at the scoring layer (random embeddings), invisible to 90%+ of users (no mobile feed), and incapable of accepting the personalization inputs it claims to use (no preferences UI). The document is unusually honest about these gaps, but then contradicts itself by labeling the system "fully implemented" in multiple places. The MVP gap table is the real PRD — the rest is a spec for a demo, not a product.

Critical

Scoring engine runs on random noise — 40% weight on placeholder embeddings

§4.2 FR-6, §8.3
Content embeddings are "random 384-dim vectors (placeholder)" yet the hybrid scoring pipeline weights cosine similarity at 40%. The system is making recommendations where nearly half the scoring weight is mathematically random. Every success metric is unmeasurable against noise-contaminated scores.
Fix: Reclassify FR-6 as a hard blocker. Add a feature flag to disable the feed entirely until real embeddings are computed. No recommendation feed should be user-facing until FR-6 is resolved.
Critical

Profile embeddings are also fake — hardcoded placeholder values for all users

§10, last bullet
recomputeEmbedding uses hardcoded values ("CEFR: B1, interests: [], goals: [], native: Bangla, age: 25") instead of real user data. Every user gets the same profile embedding. Combined with random content embeddings, the recommendation engine has zero personalization from embeddings on both sides.
Fix: Add to §6.2 gap table as High/Blocker. The profile embedding pipeline must read real user data before any embedding-based scoring is meaningful.
Critical

No mobile surface for 90%+ of users

§4.8, §5
The PRD acknowledges "90%+ of active users are on mobile," then reveals the feed exists only on web. §5 Non-Goals contradicts itself — first lists mobile feed as a non-goal, then says it's MVP-required. A reader scanning §5 will conclude mobile is out of scope.
Fix: Remove mobile feed from §5 entirely. It is not a non-goal; it is a blocker. The §5 entry should only cover "admin tools on mobile."
High

No preferences UI means 30% of scoring is dead weight

§4.5, FR-17
Tag overlap (20%) and content type preference (10%) require user preferences. No web or mobile UI exists. Combined with broken embeddings (40%), 70% of the scoring algorithm receives garbage or empty inputs. Only CEFR closeness (30%) works correctly.
Fix: Elevate FR-17 to blocker. Add a preferences onboarding flow to the user's first session to ensure preference data is collected proactively.
High

Success metrics cannot validate thesis — no baseline comparison

§7
SM-R1 targets "≥30% engagement" but with a broken scoring engine, hitting 30% validates nothing about the algorithm — it only proves users click on things when shown a list. No control condition exists (random feed vs. hybrid-scored feed).
Fix: Add a baseline comparison: measure engagement on a "newest content at your CEFR level" feed vs. the hybrid-scored feed. Without this, SM-R1–R4 are vanity metrics.
High

Content depletion spiral — all interactions excluded from pool

§4.3 FR-9, §10
The candidate filter excludes any interacted content. With 12 seed items (2 per CEFR level) and a ±1 filter, a B1 user has at most 6 candidates. After viewing all 6, the feed is permanently empty.
Fix: Exclude only dismiss and complete interactions. Allow viewed/liked/bookmarked items to reappear with a decay penalty. Add an explicit empty-state design with a prompt to try a different CEFR level or content type.
High

createContent is Protected, not Admin — security gap

§8.2
The API table lists createContent as "Protected" (any authenticated user), but §4.1 FR-1 says "[Admin] can create." Any learner can create content items. A malicious user could flood the library.
Fix: Gate createContent behind adminProcedure. If user-generated content is planned, make it a separate moderated endpoint.
High

Public seed endpoint is a wipe-the-database button

§4.7 FR-22, §8.2
The seed endpoint is publicProcedure with onConflictDoUpdate. Anyone can call it and overwrite content, preferences, and interactions with demo values. The PRD classifies this as Medium — it is a critical production safety issue.
Fix: Reclassify to High/Blocker. Remove the endpoint in production builds or gate behind adminProcedure + NODE_ENV !== 'production'.
Medium

Scoring weights are hardcoded with no tuning mechanism

§4.3 FR-8, §10
Weights (0.4/0.3/0.2/0.1) are "reasonable starting values" but A/B testing is a non-goal, no config endpoint exists, and no analytics correlate weight changes with engagement. The weights are hardcoded with no path to optimization.
Fix: Externalize weights to environment variables or a config table. Add a §9 Open Question about tuning methodology.
Medium

"View" and "complete" interactions undefined — SM-R4 is meaningless

§4.4
Content links to external source URLs. The system cannot know if a user actually consumed the content or just saw the card. "View" and "complete" are undefined. SM-R4 (like-to-view ratio) is meaningless if "view" means "card appeared on screen."
Fix: Define "view" (e.g., user clicked through to detail page or opened source URL) and "complete" (user explicitly marked complete). Add to §3 Glossary.
Medium

Score cache goes stale on preference update — 24h delay

§4.3 FR-10, FR-16
When a user updates preferences, cached scores (24h TTL) are not invalidated. A user who sets interests for the first time sees zero change until cache expires — contradicting the "feels like the app knows me" JTBD.
Fix: Invalidate cached scores on preference change, or auto-trigger recalculation after preference update.

Mechanical notes

  • Section order: §4.7 and §4.8 are out of order — §4.8 (Native Mobile, FR-23) appears before §4.7 (Seed & Demo, FR-22). Breaks both section and FR numbering expectations.
  • FR-4 scope ambiguity: FR-4 appears in both §6.2 (MVP gaps) and §6.3 (v2 roadmap) at Medium priority. This is contradictory — it cannot be both MVP and v2 scoped.
  • API security mismatch: createContent in §8.2 is "Protected" (authenticated user) but §4.1 FR-1 says "[Admin] can create." Likely uses protectedProcedure instead of adminProcedure — flag as security gap.
  • FR-8 signal 4 wording: Says "Binary — 0.1 if the content type matches" but 0.1 is the weight, not the signal value. Should read: "Binary signal (1.0 if match, 0.0 otherwise) × 0.1 weight."
  • Frontmatter date: updated: 2026-06-13 but no changelog or revision history explains what changed between 06-12 and 06-13.
  • No addendum.md: Confirmed absent. PRD is self-contained — this is fine.
  • FR numbering in document order: FR-1 through FR-22 exist, then FR-23 in §4.8 which appears before §4.7 (FR-22). Reading top-to-bottom, FR-23 is encountered before FR-22.
